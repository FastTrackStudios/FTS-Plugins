# Testing Specification

Requirements for the testing framework across all FTS plugins.

## Test Categories

### Unit Tests (DSP)

r[test.dsp.impulse-response]
Every filter type in fts-dsp must have an impulse response test that verifies the frequency response matches expected behavior (e.g., a 1kHz lowpass at 44.1kHz passes DC and attenuates 10kHz).

r[test.dsp.reset-state]
Every Processor implementation must have a test verifying that `reset()` returns the processor to its initial state: processing silence after reset must produce silence.

r[test.dsp.nan-free]
Every Processor must be tested with edge-case inputs (silence, DC offset, full-scale, NaN, infinity) and must never produce NaN or infinity in its output.

r[test.dsp.deterministic]
Processing the same input buffer twice through a freshly-reset processor must produce bit-identical output.

r[test.dsp.sample-rate-independence]
Filters must be tested at multiple sample rates (44100, 48000, 88200, 96000, 192000) to verify that frequency response scales correctly with sample rate.

### Algorithm Accuracy Tests

r[test.algo.reference-match]
Each ported Airwindows algorithm must be validated against a reference output generated by the original C++ code (from airwin2rack) processing the same input. Tolerance: within floating-point rounding (< 1e-10 per sample).

r[test.algo.reference-generation]
Provide an xtask command (`just generate-references`) that compiles and runs the original C++ algorithm against test signals and saves reference output as binary files.

### Profile Tests

r[test.profile.constraints-applied]
When a profile is active, verify that all its constraints are enforced: fixed params are at their fixed values, clamped params are within range, stepped params are at valid steps.

r[test.profile.compound-mapping]
For compound mappings (e.g., LA-2A Peak Reduction), verify that sweeping the control value from 0 to 1 produces the expected values on all mapped core parameters.

r[test.profile.switch-continuity]
Switching between profiles must not produce a discontinuity greater than a configurable threshold in any core parameter value (smooth interpolation test).

### Integration Tests (Plugin)

r[test.plugin.clap-load]
Each plugin must pass CLAP validation: load in a CLAP test host, instantiate, process a buffer, and verify no crashes or assertion failures.

r[test.plugin.state-roundtrip]
Save plugin state (all parameters + profile selection) → reload → verify all parameters are identical.

r[test.plugin.parameter-range]
Sweeping every exposed parameter from min to max and back must not produce NaN, infinity, or crashes.

### DAW Integration Tests (REAPER)

r[test.reaper.load-plugin]
Using the daw crate's reaper-test framework, verify each plugin loads in REAPER, appears in the FX browser, and can be inserted on a track.

r[test.reaper.automation-readback]
For offline analysis plugins (gate, trigger, rider): run analysis on a test track, verify the written automation points are readable back via AutomationService and match expected values.

r[test.reaper.offline-roundtrip]
Run offline analysis → write automation → render the track → compare rendered output against expected gated/triggered/ridden result.

### WASM Tests

r[test.wasm.compile]
All `*-dsp` and `*-profiles` crates must compile to `wasm32-unknown-unknown` in CI. Compilation failure is a blocking test failure.

r[test.wasm.process]
Run each DSP chain's process function in a WASM runtime (wasmtime) with test signals and verify output matches native output within floating-point tolerance.

## Test Infrastructure

r[test.infra.test-signals]
Provide a shared test signal library: impulse, white noise, sine sweep (20Hz-20kHz), pink noise, silence, DC offset, full-scale square wave.

r[test.infra.comparison]
Provide comparison utilities: max absolute difference, RMS difference, frequency-domain comparison (FFT magnitude comparison within tolerance).

r[test.infra.xtask-commands]
Provide xtask commands:
- `just test dsp` — run all DSP unit tests
- `just test profiles` — run all profile constraint tests
- `just test plugins` — run CLAP/VST3 integration tests
- `just test wasm` — compile and test WASM targets
- `just test reaper` — run REAPER integration tests (requires REAPER)
- `just test all` — run everything

## CI Requirements

r[test.ci.dsp-on-all-platforms]
DSP tests must run on macOS, Linux, and Windows in CI.

r[test.ci.wasm-compile]
WASM compilation check must run in CI on every PR.

r[test.ci.no-unsafe]
DSP crates must not contain `unsafe` code. If unsafe is required (e.g., for SIMD optimization), it must be isolated behind a safe API and have dedicated safety tests.
